# Phase 1: MVP Architecture

ShizenAI Phase 1 establishes the foundational data ingestion and semantic search pipelines. The goal of this phase was to construct a "Local-First" environment capable of taking unstructured text, summarizing it, vectorizing the summary, and persisting it to a database capable of native cosine similarity searches.

Below are the architectural diagrams detailing both the high-level intent and the low-level, technically rigorous data flows.

---

## 1. High-Level Architecture (The "What")

At a macro level, the system is composed of four containerized services that run locally on the developer's machine. 

The user interacts with the **Frontend**, which delegates heavy lifting to the **Backend**. The Backend acts as the orchestrator: it converses with **Ollama** to synthesize intelligence (summaries and vector math), and then persists the mathematical truth to **PostgreSQL**.

```mermaid
flowchart TD
    User([User]) -->|Input Raw Text| Frontend[React / Vite Frontend]
    
    subgraph Containerized Stack
        Frontend -->|HTTP Requests| Backend[FastAPI Backend]
        
        Backend -->|Prompt for Summary| Ollama[(Ollama Server)]
        Backend -->|Prompt for Embedding| Ollama
        
        Backend -->|Persist Payload + Vector| Postgres[(PostgreSQL + pgvector)]
    end
    
    User <-->|Search & Retrieve| Frontend
```

---

## 2. Low-Level Architecture & Data Flow (The "How")

The actual pipeline is significantly stricter regarding type coercion and dimensional guarantees. The sequence relies heavily on Pydantic validation on the FastAPI boundary, and strict `Vector(768)` dimensions natively inside Postgres.

### Data Types & Dimensions
- **Text Payload**: Stored as native `TEXT` inside Postgres.
- **Summary**: Stored as native `TEXT`. Generated by `llama3`.
- **Embedding Dimensions**: `768-dimensional float arrays`, natively emitted by `nomic-embed-text` and asserted at the database layer via `$ embedding = Column(Vector(768), nullable=False)`.

### Ingestion Sequence Diagram

```mermaid
sequenceDiagram
    participant UI as React Client (App.tsx)
    participant API as FastAPI (main.py)
    participant DB as Postgres (pgvector)
    participant LLM as Ollama (OpenAI API Compatible)

    UI->>API: HTTP POST /api/v1/ingest (IngestRequest {text: string})
    activate API
    
    Note over API, LLM: 1. Summarization Phase
    API->>LLM: HTTP POST /v1/chat/completions (model: "llama3", temp: 0.3)
    activate LLM
    LLM-->>API: 200 OK (choices[0].message.content: string)
    deactivate LLM
    
    Note over API, LLM: 2. Embedding Phase
    API->>LLM: HTTP POST /v1/embeddings (model: "nomic-embed-text", input: summary_string)
    activate LLM
    LLM-->>API: 200 OK (data[0].embedding: float[768] array)
    deactivate LLM
    
    Note over API, DB: 3. Persistence Phase (Relational + Semantic)
    API->>DB: INSERT INTO knowledge_records (raw_text, summary, embedding)<br/>VALUES (Text, Text, Vector(768))
    activate DB
    DB-->>API: TRANSACTION COMMIT (KnowledgeRecord {id: UUID, created_at: DateTime})
    deactivate DB
    
    API-->>UI: HTTP 200 OK (IngestResponse {status, record_id, summary})
    deactivate API
```

### Retrieval Sequence Diagram

```mermaid
sequenceDiagram
    participant UI as React Client
    participant API as FastAPI
    participant DB as Postgres
    participant LLM as Ollama

    UI->>API: HTTP POST /api/v1/search (SearchRequest {query: string, limit: int})
    activate API
    
    API->>LLM: HTTP POST /v1/embeddings (model: "nomic-embed-text", input: query)
    activate LLM
    LLM-->>API: 200 OK (query_embedding: float[768] array)
    deactivate LLM
    
    Note over API, DB: Cosine Distance Operator: <br/> embedding <=> query_embedding
    API->>DB: SELECT id, summary, 1 - (embedding <=> query_embedding) AS similarity_score <br/>ORDER BY distance LIMIT 5
    activate DB
    DB-->>API: Returns list of records and float distances
    deactivate DB
    
    API-->>UI: HTTP 200 OK (SearchResponse {results: SearchResultItem[]})
    deactivate API
```
